## Approaches considered

- **A — Typed enum on `SessionCreateRequest` (deep, recommended):** Add `LaunchMode = 'default' | 'bypass-permissions' | 'agents'` (string-literal union) and a new optional `launchMode: LaunchMode` field on `SessionCreateRequest`. Renderer's `NewSessionDialog` picker writes this. `ClaudeProvider` and `CLIManager` switch on the enum to construct the command argv. The boolean runtime branches at `cli-manager.ts:353` and `claude-provider.ts:38` are deleted — the launch mode is the single source of truth. The `bypassPermissions` setting is read once at dialog-open time only, to pick the picker's *default*; per-session selection wins from then on. Codex provider ignores the field (optional, only consumed in the Claude path).
- **B — Raw command-args passthrough (shallow):** Add `commandArgs?: string[]` to `SessionCreateRequest`. The renderer constructs the argv (`['--dangerously-skip-permissions']` or `['agents']` or `[]`) directly; main appends whatever it receives to `claude` and spawns. The picker maps user selection to one of these arrays in renderer code.
- **Tradeoff axis:** **domain enum vs. command-args passthrough** (declarative vs. imperative). Equivalently: who owns argv construction — main process (A) or renderer (B)?
- **Chosen:** **A** — the enum makes "unknown mode" impossible to express at the IPC boundary (TypeScript narrows three string literals); future modes are enumerable and easy to add a switch case for; the same mapping-from-label-to-argv exists in *either* approach (B just relocates it to the renderer), so the "shallowness" of B is an illusion. A also keeps command construction co-located with everything else that touches the spawned process (e.g. environment, cwd, IProvider).

### Depth analysis (Ousterhout principle 1)
- **A's interface** to the renderer is one enum field. Internally the Claude path hides argv construction, version probing, availability gating, env propagation, and `IProvider` dispatch. Implementation >> interface — **deep**.
- **B's interface** is a string array the renderer constructs and main passes through. Implementation ≈ interface — **shallow**. It also leaks command-line shape into the renderer, where it doesn't belong (renderer can't safely know which flag names exist on which CLI version).

### Error cases enumerated (Ousterhout principle 2)
**Approach A must handle:**
1. **Unknown `launchMode` at the IPC boundary** → TypeScript narrows; the consumer's `switch` has a `default` case that warns and falls back to `'default'`. → *Mostly defined out of existence at compile time; runtime fallback handles a downgraded-CLI edge case.*
2. **User selects `'agents'` while it's unavailable** (e.g. picker bug or stale availability cache) → main-side defensive check before spawn: if `getCachedAvailability().status === 'unavailable'` and `launchMode === 'agents'`, fall back to `'default'` and surface a warning toast. (Picker disables the option, so this is a defense-in-depth check only.)
3. **`bypassPermissions` setting unset / missing** → default value `false` → picker default `'default'`. → *Defined out of existence by the setting reader's typed default.*
4. **Probe fails / times out** → availability is `'cli-not-found'` or `'version-unparseable'` → picker disables `agents`. → *Surfaced as a typed `AgentViewUnavailable`, not as a thrown error.*
5. **Codex (non-Claude) session sent with `launchMode`** → field is optional and Codex provider doesn't read it. → *Field is structurally inert for non-Claude providers; no special-casing needed.*
6. **Hot-swap: user changes the `bypassPermissions` setting while `NewSessionDialog` is open** → dialog's initial default is already loaded; user can manually flip the picker if they want. Acceptable as-is; document the trade-off in `Notes`.
7. **Shell injection** → mode → argv mapping is a fixed-shape `switch`; no user-controlled string interpolation. → *Defined out of existence.*

**Approach B would additionally have to handle:** unknown / unsafe argv strings, version-skew (renderer constructing args for a CLI version that doesn't support them), and the same map-of-label-to-args that A has — just relocated.

### Generality / plausible callers (Ousterhout principle 3)
- **Today:** `NewSessionDialog` picker writes `launchMode` into `SessionCreateRequest`.
- **Plausible future caller #1:** "Duplicate session" command — opens a new session with the same `launchMode` as the source.
- **Plausible future caller #2:** right-click "Open in launch mode X" on an existing session.
- All three use the same enum on the same request shape; no special-casing.

### Module layout
- `LaunchMode` type → add to `src/shared/ipc-types.ts` next to `SessionCreateRequest` (the natural home; it's session-creation vocabulary).
- New `useAgentViewAvailability` renderer hook → `src/renderer/hooks/useAgentViewAvailability.ts` (tiny — calls the IPC method once on mount, returns the result).
- New IPC method `agentView:availability` (invoke, returns `AgentViewAvailability`) — adds one entry to `IPCContractMap`. The cached value lives in `src/main/index.ts` and is built once at the existing delayed-init block.
- Picker UI → inline inside `NewSessionDialog.tsx`. If markup grows past ~30 lines extract to `src/renderer/components/ui/LaunchModePicker.tsx`.
- Main-process consumers — `src/main/cli-manager.ts:353` and `src/main/providers/claude-provider.ts:38` lose their `if (bypassPermissions)` branches, gain a `switch (request.launchMode ?? defaultMode)` driven by the field.
